
Descripción de la actividad
Finalizando la asignatura en esta semana 8, deberás hacer la actividad sumativa de la
Experiencia 3, llamada "Consumiendo Stream de datos", de manera grupal (2 estudiantes),
donde tendrán que construir un componente de Back End que se conecte a un Stream de
datos para poder obtener información según el requerimiento, el caso será el mismo que se
ha desarrollado en las actividades sumativas anteriores. Finalmente, como parte dos
deberán hacer un video de presentación de lo realizado.
Instrucciones específicas
Parte I: Documentación
1. Deberán entregar en un archivo comprimido (.rar o .zip) los componentes creados. El
comprimido deberá contener el código fuente de la aplicación.
2. El cluster de Kafka debe estar configurado en 3 contenedores, mas 3 contenedores de
zookeeper y uno con kafka-ui, ademas hay que considerar que hay que levantar tanto
la cola como los microservicios utilizando docker-compose
3. Los microservicios creados deben cumplir con los siguientes aspectos:
• Deben ser desarrollados bajo el Framework Spring con spring boot.
• Manejo y organización del utilizando herramientas colaborativas y de repositorios
en GIT.
• Los microservicios deben usar una conexión a una Base de Datos en Oracle donde
estén construidas sus respectivas tablas para manipular su información.
• Debe implementar todos los controladores necesarios para manipular las comunicaciones
con RESTful (get,post,put,delete).
• Debe manipular POSTMAN para poder validar el correcto funcionamiento de las consultas.
• También adjunta en el ZIP este formato de respuesta.
Para la creación del sistema deberán tener en cuenta lo siguiente en el caso asignado:
Caso
“Desarrollar un sistema para la gestión y monitorización en tiempo real del transporte público,
incluyendo ubicaciones de vehículos y horarios”. En esta tercera experiencia nos corresponde
construir las piezas del backend relacionadas al sistema de monitoreo del paciente, por lo que
tendremos un cluster Kafka y varios microservicios.
Funcionalidades clave
Tarea 1: Configuración del clúster Kafka
1. Configurar un clúster Kafka:
• Configura un clúster Kafka en la nube con al menos tres nodos.
• Asegúrate de que el clúster sea accesible y esté corriendo.
• Crear un tópico para las señales vitales:
• Crea un tópico llamado “ubicaciones_vehiculos” donde se publicarán las
actualizaciones de ubicación de los vehículos.
• Crea un tópico llamado “horarios” donde se publicarán las actualizaciones de los
horarios de los vehículos.
Tarea 2: Implementación de microservicios
Microservicio productor de ubicaciones:
• Desarrolla un microservicio que registre las actualizaciones de ubicación de los
vehículos y las publique en el tópico “ubicaciones_vehiculos” de Kafka.
• Asegúrate de que las ubicaciones se registren en tiempo real.
4
Microservicio de Procesamiento de Señales:
• Desarrolla un microservicio que consuma mensajes del tópico
“ubicaciones_vehiculos” de Kafka.
• Implementa lógica para actualizar los horarios basado en las ubicaciones de los
vehículos.
• Publica las actualizaciones de horarios en el tópico “horarios” de Kafka.
Microservicio de Monitorización:
• Desarrolla un microservicio que lea las actualizaciones de horarios y ubicaciones
desde los tópicos “horarios” y “ubicaciones_vehiculos” en Kafka.
• Este microservicio debe guardar en una base de datos Oracle Cloud el registro
de las ubicaciones y horarios de los vehículos, generando un resumen de
ubicaciones en la base de datos al final del día, mostrando a nivel general los
lugares visitados y horarios en los que se llegó al lugar.
Detalles Técnicos:
• Cluster Kafka: se configurará un cluster Kafka sobre docker lab, el cual debe
tener 3 nodos de kafka, 3 nodos de zookeeper y un nodo de kafka-ui para la
interfaz gráfica.
• Persistencia en Oracle Cloud: se utilizará una base de datos Oracle Cloud para
almacenar los datos de las alertas. El microservicio correspondiente usará
Spring Data JPA o Hibernate para la interacción con la base de datos.
• Productor de ubicaciones: un microservicio productor utilizará un cron para
poder generar la información necesaria cada un segundo.
• Deberán usar Docker para su desarrollo y deben tomar en consideración al
momento de probar su desarrollo final cambiar las urls de las APIs de
comunicación a las que genera Docker Lab una vez que los hayan subido y
puesto a funcionar en el mismo
